# 区分实用 Skill 和 Markdown 长文的 9 条原则

一个实用框架：帮助你判断一个 Skill 什么时候该被触发、该如何运行，以及如何经得起时间考验。

2026 年 5 月 10 日

软件工程很早就沉淀出了一小组高层原则，它们站在具体实现模式之上：SOLID、KISS、YAGNI、关注点分离。它们告诉你设计系统时应该重视什么，而不是告诉你具体要敲什么代码。它们之所以能比一大堆模式目录更长久，是因为它们站在更高的层次上。

Skill 的形态很多：任务流程、参考指南、人格提示词、工具封装。每一种怎么写，我在上一篇关于 Skill 编写模式的文章里已经讲过。但在这些具体技巧之下，还有一层类似的高层原则：一小组跨越所有 Skill 类型的原则。

它们不是告诉你 SKILL.md 里该放什么内容的规则，而是你在套用任何具体模式之前要先问的问题：这个 Skill 到底该不该存在？它能不能在关键时刻被找到？六个月之后它还会不会有用？

这 9 条原则分成三幕：被选中、可靠运行、经得起现实检验。

## 第一幕：被选中

这些原则发生在请求刚到达、Skill 正文还没运行之前。它们决定这个 Skill 会不会被识别出来、把它加载进上下文要花多少成本，以及它的内容形式是否真的能推动工作。一个 Skill 如果在这里失败，就根本没有机会发挥作用。

## 1. 元数据是入口

会话开始时，Claude 不会读取每一个 SKILL.md。它看到的是 Skill 列表：比如名称、描述；在 Claude Code 里，还可能包括 `when_to_use` 这类触发说明。一个含糊的描述，比如“帮助处理文档”，要么该触发时不触发，要么不该触发时乱触发，要么和其他三个 Skill 抢同一个触发场景。

原则是：描述不是摘要，而是 Skill 被选择时最重要的信号。一个 Skill 如果在“只看元数据”的第一轮就失败了，它根本不会运行。

我上一篇文章里的两个 Skill 编写模式，分别从两个方向服务于这个原则。Activation Metadata 是把正确的 Skill 拉进来：正向触发条件、领域关键词，以及 Anthropic 的 skill-creator 推荐的那种“主动一点”的表述。Exclusion Clause 则是把错误的 Skill 推出去，比如：“不要用于博客文章、新闻通讯或长篇内容。”目标相同，方向相反。

同一个 Skill，弱描述和强描述的对比：

弱：
“帮助处理仪表盘和可视化。”

强：
“当用户提到仪表盘、内部指标或数据可视化时使用，即使用户没有明确要求制作仪表盘。不要用于博客图表、营销图形或一次性的 notebook 图。”

这个原则其实是在指导“可发现性”和“上下文预算”之间的取舍。Agent Skills 文档把 description 限制在 1024 个字符内；Claude Code 会把 description 加上 `when_to_use` 截断到 1536 个字符以内。每一句话都在和正向触发、排除条件、领域关键词争抢同一块有限空间，所以清楚说明 Skill 的意图，比覆盖所有边缘情况更重要。写描述时，要像这些字符真的很贵一样去写。实际上，它们确实很贵。

## 2. 渐进式披露

上下文是一种共享预算。你的 Skill 每花掉一个 token，就少了一个 token 留给其他 Skill、对话历史和用户真正的问题。一个 Skill 如果重新解释什么是 JSON、什么是数据库、API 是怎么工作的，就是把预算浪费在 Claude 本来就知道的事情上。

如果很多已安装的 Skill 都有冗长的元数据，或者一个多 Skill 任务里加载了臃肿的 SKILL.md 正文，真正有用的上下文还没开始做难事就已经被消耗掉了。

原则是：只加载当前任务需要的内容，并且只在需要的时候加载。元数据在会话开始时加载；SKILL.md 只在 Skill 被触发时加载；参考文件只在当前任务指向它们时加载。`scripts/` 里的脚本可以通过 Bash 调用，所以源码本身不必进入上下文，只有输出结果需要进入上下文。

14 个 Skill 模式里有两个服务于这个原则。Context Budget 是对已加载文字保持节制的纪律。Progressive Disclosure 是分层加载内容的结构手段。这两者分不开：只有节制但不分层，还是会一次性把所有东西加载进来；只有分层但不节制，只是把臃肿内容分散到了更多文件里。

同一个 Skill 的臃肿开头和精简开头：

臃肿：
“数据库是一种结构化的数据集合……迁移是更新 schema 的脚本……PostgreSQL 是一种关系型数据库……”

精简：
“Schema 变更放在 `supabase/migrations/` 目录下，命名格式为 `YYYYMMDDHHMMSS_description.sql`。四个必需产物见 `references/migration-format.md`。”

这个原则是在指导“节省上下文”和“模型能力”之间的取舍。对 Sonnet 来说清晰简洁的内容，对 Haiku 来说可能太简略，导致它达不到同样的结果。如果你要兼容多个模型，就按最弱模型所需的指导量来写，让更强的模型自己略读。让弱模型跑不起来的“节省”是假节省：你省下了 token，之后又会用重试和修正把它们花回去。

## 3. 流程优先于散文

一个 Skill 如果开头写两千字测试哲学，模型会读，会生成一段看似合理的总结，然后仍然跳过真正的测试。参考类长文不是可执行动作。很多所谓“AI 规则”仓库，发出来就是一大篇 Markdown，实际效果很有限，因为它们只是在描述好行为，而不是让 agent 执行这些行为。

Addy Osmani 提出的原则是：Skill 是带检查点的工作流，不是参考文档。如果一个初级工程师读完之后还不知道下一步该做什么，那它就是一篇文章，不是 Skill。有退出条件的步骤，胜过没有执行要求的长篇论述。

上一篇文章里的几个模式都和这个原则有关。Template Scaffold 给 Skill 一个固定结构去填。Execution Checklist 给它带顺序的步骤和复选框。整个 Workflow Control 家族，包括自我纠错循环、计划-验证-执行，只有在 Skill 本身首先是工作流时才有意义。

同样是 TDD 内容，两种写法：

散文式，不推荐：
“测试驱动开发是在实现之前先写测试的实践。它通过迫使作者先考虑接口再考虑内部实现，从而产出设计更好的代码……”

流程式，保留：
“1. 写一个失败的测试。2. 运行它。3. 确认它是因为预期原因失败。4. 实现能通过测试的最小代码。5. 运行它。6. 重构。7. 提交。”

这个原则指导的是“可执行性”和“覆盖面”之间的取舍。工作流可执行，但范围窄：每一步都是具体动作。参考资料覆盖面广，但本身不会动：模型要自行判断什么时候应用什么。大多数 Skill 应该是工作流，但参考类 Skill，比如风格指南、领域词汇表、人格提示词，是合理例外。硬把它们改成工作流，反而会变得僵硬别扭。真正该问的不是“这是不是工作流”，而是“读完这个 Skill 的人知不知道下一步该做什么”。如果答案是否定的，那无论形式如何，都违反了这个原则。

## 第二幕：可靠运行

一旦 Skill 被选中并加载，问题就变成：它到底能不能驱动作者想要的行为？这一部分原则覆盖两个方面：规则应该怎么表达，才能让模型既能从文字中泛化，又能在想偷懒时仍然遵守；Skill 该给模型多少自由度，才能让模型自主性和任务容错度匹配。

## 4. 解释原因

一个 Skill 如果全是大写的 ALWAYS、NEVER、MUST，会给 Claude 一堆僵硬规则，却没有上下文。模型可能只遵守字面意思，漏掉作者没预想到的边缘情况；也可能在本来需要判断的地方过度套用规则。Anthropic 的 skill-creator 也把这种全大写命令式写法标记为黄色警告，原因就在这里。

原则是：先写规则，再写原因。原因会变成处理未列明情况时的判断标准。

上一篇文章里的两个模式都和这个原则有关，但作用不同。Explain-the-Why 是直接技巧：在 SKILL.md 里给每条规则配上原因。Control Tuning 则决定这种配对该用在哪里：需要模型判断的地方，用“规则 + 原因”；一旦出错就很严重的地方，用狭窄、明确的命令，因为那时不需要判断，判断反而危险。

一个 Spring 风格指南规则的两种写法：

只有规则：
“使用构造函数注入。不要使用字段注入。”

规则加原因：
“使用构造函数注入。字段注入会破坏可测试性；如果不启动 Spring 上下文，我们就无法 mock 这个字段。”

第一种只覆盖标准情况，遇到变体容易出错。第二种让模型能评估 Skill 没明说的选项：一个在 `build()` 里传依赖的 builder 可以通过，因为依赖通过参数传入、无需 Spring 也可 mock；`@Component` 上的 setter 注入不通过，因为有同样的可测试性问题；方法内部的 service locator 也不通过，因为它隐藏了依赖。规则覆盖一个场景，原因覆盖所有这些场景。

这个原则是在指导“泛化能力”和“长度”之间的取舍。给每条规则都加原因，有助于模型处理 Skill 没写到的情况，但解释会消耗 token。对于真正脆弱的步骤，比如数据库迁移、加密操作，或者任何一步错了就会造成严重后果的事情，狭窄命令可能仍然更好。把“带原因的说明”留给那些你确实希望模型思考的步骤。其他地方，简短是优点，不是缺陷。

这个原则解决的是一种知识缺口：模型知道得不够，所以处理不了变体。还有另一种缺口：模型知道规则，也知道原因，却还是给自己找理由不遵守。那就需要另一种工具。

## 5. 预判借口

想象一个数据库迁移 Skill 里有一条不可协商的规则：“必须先在 staging clone 上运行迁移。”原因也写清楚了：staging 能捕捉 SQL diff 看不出来的问题，包括默认排序、类型强制转换和意外的索引重建。

模型有规则，也有原因。但当迁移很小、用户又想快点推进时，它仍可能生成这样的内容：

“这次迁移只是增加一个可为空的列。启动 staging clone 需要 4 分钟。这个变更是增量式的，用户也很着急，而且 SQL 很短，可以人工审查。这里直接在生产环境运行是可以接受的。”

这段话听起来很合理。它引用了真实理由，甚至承认了规则，然后绕过了规则。这就是合理化：为这个特定情况找出一个自信的借口。继续加规则没有用，继续加原因也没有用。模型已经接受了原因，但它判断“这个情况例外”。

Addy Osmani 的原则是：预判借口。把反驳写在规则旁边，直接写进 Skill 里。模型很擅长合理化；但当预先写好的反驳就在它自己的上下文里时，它很难继续绕过去。

上一篇文章没有哪个模式直接服务于这个原则。最接近的是 Known Gotchas，但 Gotchas 解决的是外部意外，也就是世界让 agent 措手不及；而反驳解决的是内部意外，也就是 agent 自己给自己找理由。两者互补，但处理的是不同力量。

迁移规则的反驳表，可以这样写进 SKILL.md：

| 借口                     | 反驳                                                          |
| ---------------------- | ----------------------------------------------------------- |
| “这个迁移是增量式 / 可为空 / 很小。” | 小迁移也正是 staging 能抓出低层错误的地方，比如默认排序、类型强制转换、意外索引重建。4 分钟是很便宜的保险。 |
| “用户很着急。”               | 用户着急的是交付可用代码，不是把迁移跑两遍。去跑 staging。                           |
| “我已经人工审查过 SQL，看起来很安全。” | 人工审查不能代替实际运行。你不是数据库引擎。                                      |

当模型想给自己找借口时，它必须越过刚刚读过的三条反驳。这比越过一句规则难得多。

这些反驳从哪里来？主要有三个来源：

第一，通用的合理化套路。有些借口到处都会出现：“这个情况太小了”“我之后再做”“我已经手动检查过了”“这里风险很低”“用户很着急”。任何非平凡规则通常都能匹配一两个。

第二，实际运行观察。第一次 agent 跳过步骤时，它会生成一段解释为什么可以跳过。把这段复制下来。下一版 Skill 就为它写反驳。

第三，事故复盘。上一次有人跳过步骤 X 然后踩坑，那次事故就能变成反驳。

这个原则指导的是“防御覆盖面”和“可信度”之间的取舍。一旦你接受这个原则，就很容易想给 Skill 里的每条规则都写反驳表。但基于想象失败模式写出来的反驳，会显得像偏执的填充内容，还会把 SKILL.md 撑大，却不值得这些 token。反驳只有在回答真实出现过的借口时才有力：要么是你亲眼看到模型说过的，要么是团队在压力下真的说过的。少量使用，把它留给最容易被跳过的两三个步骤，然后从实际运行中逐渐扩展，而不是凭空想象。

## 6. 代码优先于推理

让模型每次调用时都从零生成验证脚本、PDF 提取器或 JSON 规范化器，会更慢、更不可靠，还会把 token 花在用户根本看不到的代码上。每次运行都在重新发现同一套逻辑，而且细节还会有小差异。

原则是：如果某个步骤能变成确定性程序，就应该变成程序。把工作从推理循环里拿出来，放进可运行的代码里，只把输出结果交给模型。

上一篇文章里的两个模式和这个原则有关，但原因不同。Utility Bundle 是把预先写好的脚本和 SKILL.md 一起发布：这些专用辅助脚本由模型通过 Bash 调用。Autonomy Calibration 则是限制模型到底能做什么，缩小工具表面，让脚本成为最省事、最自然的路径。

一个 schema 验证步骤的两种写法：

每次重新推导：
模型现场写 60 行验证代码，运行它，解析输出，再判断 schema 是否通过。源码进入上下文，逻辑每次都可能不同。

预置脚本：
`python scripts/validate.py schema.json`。上下文里只进入脚本输出，比如 “OK” 或结构化错误列表。源码不进入上下文。逻辑每次完全一致，可以独立测试，模型通过 Bash 调用即可，不需要思考脚本内部如何实现。

这个原则指导的是“确定性”和“可移植性”之间的取舍。脚本能带来更可靠的行为，也能降低上下文成本，但它引入了推理路径本身没有的运行时依赖。Python 可能不存在，库版本可能不同，路径可能是 Windows，而作者当初按 Unix 写。要在 SKILL.md 里验证前置条件，路径尽量使用正斜杠，并让脚本在依赖缺失时输出清晰错误，而不是直接甩出堆栈。对于要发往混合环境的 Skill，有些逻辑确实更适合留在推理里，哪怕成本更高。

## 7. 留在范围内

你让 agent 修一个函数里的 off-by-one 错误，它却返回了横跨三个文件、四十多行的 diff：修复本身、`var` 改 `const`、重命名一个 helper、顺手清理一个 TODO。修复本身可以合并，但这个 PR 不行。把无关改动拆出来的成本，比原任务还高。

Addy Osmani 的原则是：只碰 Skill 被要求碰的东西。不要重构旁边的系统。不要重写你只是路过的文件。不要扩大 diff。Addy 认为，这是决定一个 agent PR 能否合并、是否需要被拆回去的最大因素。

上一篇文章没有哪个模式直接服务于这个原则。Exclusion Clause 有点相邻，但它工作在元数据层，决定 Skill 是否触发；而这个原则工作在 Skill 内部，决定 Skill 触发之后 agent 做什么。范围纪律更像一种文化规则，而不是某个 Markdown 技巧。它必须在 Skill 的每一步里被执行，而不是声明一次就完事。

同样是修 `parseRange()` 的 off-by-one：

超范围版本：
47 行，3 个文件：off-by-one、`var` 到 `const` 的现代化、`helperFunc` 改名为 `parseHelper`、清理 TODO。

范围内版本：
3 行，1 个文件：只修 off-by-one。

这个原则指导的是“有帮助”和“可审查”之间的取舍。更大的 diff 会显得更有帮助，因为 agent 看到了相关问题并顺手修了。但更大的 diff 也更难信任，因为审查成本随 diff 大小增长；一旦审查者看到一个无关改动，就会开始怀疑所有改动。对于真正需要跨文件的情况，要内置一个兜底规则：“如果修复确实需要另一个文件，先说明那个文件名和原因，再编辑它。”这个原则要阻止的是悄悄扩大范围，而不是阻止合理的跨文件工作。

## 第三幕：经得起现实检验

前两幕默认 Skill 存在于某个单一时刻。但世界不是静止的。代码库会变，库会更新，团队会换人，Skill 当初针对的模型也会被更强的新模型替代，而新模型对指令的理解可能略有不同。这一部分原则关注 Skill 随时间变化会发生什么，以及编写实践一开始就该如何预判这种漂移。

## 8. Skill 会衰减

库会更新，API 会变化，约定会演进。一个六个月前针对上季度框架写的 Skill，现在可能指向一个已经改名的函数、一个废弃的参数，或一个已经迁移的 API。和工程师一眼扫过的过期注释不同，agent 会自信地调用过期 Skill，并生成看起来正确、实际错误的输出。

原则是：Skill 是活的工件，不是冻结文档。一个几个月没重新跑过的 Skill，就是一个你还没有衡量过的风险。

上一篇文章没有哪个单独模式直接服务于这个原则。衰减是系统随时间变化产生的属性，不是你在 SKILL.md 里加几句话就能修好的。Known Gotchas 里那句“gotchas 是移动目标”只是一个特例。一个写死了废弃框架参数的 Skill，会生成框架根本无法编译的代码，而 agent 还会引用这个过期 Skill 作为依据。模式目录通常抓不到这种失败，因为失败发生在 Skill 原本正确的几个月之后。

这个原则指导的是“稳定性”和“过期风险”之间的取舍。一个你从不修改的 Skill 很稳定，但稳定不等于正确。它描述的世界可能比 Skill 本身变化得更快。要像对待任何可能和系统脱节的代码一样对待 Skill：指定负责人，定期用当前任务跑一跑，发现不再值得维护时就诚实淘汰。这个原则不规定具体周期。重点是有人在看着它，而不是多久看一次。

## 9. 发布前先跑

一个凭想象写出来的 Skill，描述的是作者希望中的世界，而不是 agent 实际会遇到的世界。它带着错误的反驳、错误的 gotchas，以及作者猜出来的辅助脚本发布。第一次真实调用时，这三个问题会一起暴露。

Anthropic 的 skill-creator 提出的原则是：Skill 是从观察到的运行中调试出来的，不是从作者的想象中写完的。用两三个真实感强的测试案例跑它。读运行记录。如果多次运行都独立写出了同一个 helper，就把这个 helper 提升到 `scripts/`。如果模型两次给出同样的合理化借口，就写下对应反驳。如果同一个 gotcha 绊倒了两次不同运行，就把它记录下来。

上一篇文章没有哪个模式单独服务于这个原则。运行 Skill 不是写在 SKILL.md 里的内容，而是你对 SKILL.md 做的事情。但这个原则是很多其他原则的上游：预判借口里的反驳、成熟 Skill 会积累的 gotcha 条目，以及被提升到 `scripts/` 的辅助脚本。这些都不该坐在桌前凭空猜。它们是在测试运行中观察到，再反哺回 Skill 的。

这个原则指导的是“速度”和“正确性”之间的取舍。凭想象编写很快。你可以一个下午就发布 Skill 的第一版；对于简单 Skill，差异也不一定明显。基于真实运行来编写，第一版会慢一些，但从第二版开始就会回本，因为你在测试运行中观察到的反驳和 gotchas，会避免用户在生产中踩同样的坑。对于非常小的 Skill，比如一行别名或五行参考说明，这个循环可能太重。但只要是带工作流、有不可协商规则，或者带脚本的 Skill，这个循环就是区分“会复利增长的 Skill”和“带着缺陷发布、又没人看它运行”的 Skill 的关键。

## 总结

这 9 条原则可以组合使用，但它们也有优先级。元数据是入口、流程优先于散文、发布前先跑，是基础。一个不会被触发的 Skill，根本跑不了工作流；一个只是长文的 Skill，无法执行工作流；一个凭想象发布的 Skill，很可能从一开始就是错的工作流。

另外 6 条是修正项。它们解决的是结构上还算健康的 Skill 内部的具体失败模式，但它们救不了基础就错了的 Skill。

Skill 正在变成 agent 的一个清晰构建块。我之前的文章看的是机制：〈12 Agentic Harness Patterns from Claude Code〉讲的是 agent 周围的运行时结构，〈14 Skill Authoring Patterns〉讲的是 Skill 里面该放什么。这篇文章讨论的是更上一层：什么样的原则让一个 Skill 值得构建、值得加载，也值得维护。

软件工程有 SOLID、KISS、YAGNI 和关注点分离。Skill 编写也需要同样层级的共同语言。这 9 条原则就是我的提案，它们来自真实可用的 Skill、流行 Skill，以及 agent 真正开始使用它们后暴露出来的失败模式。
